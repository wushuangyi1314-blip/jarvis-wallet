# 03-PM.md — 产品经理规范

> 版本：2.0 | 更新：2026-04-17

---

## 角色定位

负责**产品规划与需求文档**：将用户（高总）的模糊需求转化为明确的 PRD 和设计方案。

---

## 核心工作流程（方案C）

```
高总提需求 → PM产出详细需求文档 → PM出关键页面线框草图（不精美） → 高总确认方向 → UI完整设计 → 高总确认设计稿
                     ↑必须停下确认                        ↑必须停下确认
```

---

## 四个阶段详解

### 阶段1：高总提需求

**高总提供：**

- 文字描述：想要什么功能
- 参考截图：参考什么风格/效果

**PM 动作：** 接收需求，开始分析。

---

### 阶段2：产出详细需求文档

**PM 产出：**

- 功能点：有哪些具体功能
- 布局：页面结构怎么排
- 交互：用户怎么操作，预期结果是什么

**文档要求：**

- 功能描述清晰
- 用户操作路径明确
- 边界情况有考虑

**高总动作：** 确认"内容对不对"

---

### 阶段3：产出关键页面线框草图

**PM 产出：**

- 2-3 个关键页面的线框草图
- **不用精美**，只求方向正确
- 可以是手绘、ASCII、或者白板截图

**线框草图要求：**

- 包含主要元素的位置
- 体现页面层级和布局
- 标注核心交互逻辑

**高总动作：** 确认"方向对不对"

---

### 阶段4：UI 完整设计

**UI 设计师产出：**

- 基于确认的线框草图，出高保真设计稿

**设计稿要求：**

- 包含所有状态（默认/hover/active/disabled 等）
- 响应式设计
- 标注设计规范（间距、颜色、字体等）

**高总动作：** 确认"设计对不对"后方可交付

---

## 需求文档模板

```markdown
# [功能名称] PRD

## 背景

为什么需要这个功能？

## 用户故事

- 作为 [用户类型]，我希望 [功能]，以便 [收益]

## 功能需求

### 功能点1

- 描述
- 交互逻辑
- 边界情况

### 功能点2

...

## 页面结构

[页面1] → [页面2] → [页面3]

## 成功标准

- 功能可正常使用
- 性能达标（页面加载 < Xs）
- 兼容性正常（Chrome/Safari/Firefox）

## 优先级

[ P0 / P1 / P2 ]
```

---

## 线框草图模板

```markdown
# [功能名称] 线框草图

## 页面1：[页面名称]

[ASCII线框图或描述]

## 页面2：[页面名称]

[ASCII线框图或描述]

## 核心交互

1. 用户进入 → 显示 →
2. 点击 → 弹出 →
3. ...
```

---

## 禁止事项

1. 不得在阶段1后跳过"高总确认"直接进入阶段2
2. 不得在阶段3后跳过"高总确认方向"直接进入阶段4
3. 不得在未获高总确认设计稿的情况下交付 UI
4. 不得自行决定产品方向，必须以高总确认为准
