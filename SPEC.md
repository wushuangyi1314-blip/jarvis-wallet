# SubAgent Worker 傀儡体系执行规范

> 版本：v1.0.0 | 最后更新：2026-05-14 23:46 by 阿呆

---

## 一、核心原则

| 原则 | 说明 |
|------|------|
| **精简** | 阿呆管理成本低，流程简单 |
| **隔离** | 每个 Worker 有独立记忆文件，不冲突 |
| **有限记忆** | 记忆长度限制，主要目的是保留专业能力 |
| **阿呆主导** | 阿呆是协调者，Worker 是执行傀儡，Worker 之间互不知晓 |

---

## 二、目录结构

```
/root/.openclaw/workspace/workers/
  rd-worker/           ← 研发执行者
      definition.md     ← 角色定义 + 行为准则（永久）
      log.md           ← 工作日志（长度限制）
  ops-worker/          ← 运营执行者
      definition.md
      log.md
  pm-worker/           ← 产品执行者
      definition.md
      log.md
  [其他Worker/]
```

**注意：**
- workers/ 目录与 agents/（角色文档）完全隔离
- 小写文件名避免与阿呆的 SOUL.md、MEMORY.md 混淆

---

## 三、文件格式规范

### 3.1 definition.md（角色定义）

```markdown
# {role}-worker 角色定义

> 版本：v1.0.0 | 最后更新：YYYY-MM-DD HH:MM by 阿呆

## 核心职责
- 列出该 Worker 的主要职责

## 工作原则
1. 执行原则1
2. 执行原则2

## 说话风格
- 风格描述

## 能力范围
- 能力1
- 能力2

## 禁止行为
- 禁止1
- 禁止2
```

### 3.2 log.md（工作日志）

```markdown
# {role}-worker 工作日志

> 最后更新：YYYY-MM-DD HH:MM by 阿呆

## 核心能力（永久保留）
- 能力1
- 能力2

## 完成任务记录（最多20条，超出归档）
- YYYY-MM-DD HH:MM: 任务描述

## 历史归档
[超出20条的早期记录]
```

---

## 四、记忆长度限制

| 字段 | 限制 | 策略 |
|------|------|------|
| 核心能力 | 无限制 | 永久保留 |
| 完成任务记录 | 最多 20 条 | 超出则归档到历史 |
| 单条记录 | 最多 200 字 | 压缩摘要 |

**触发压缩条件：**
- 记录超过 20 条 → 保留最新 15 条，早期 5 条归档
- 单条超过 200 字 → 阿呆手动压缩
- log.md 总字数超过 3000 → 触发压缩流程

## 五、阿呆调用流程

### 5.1 标准 Spawn 流程

1. 阿呆接收任务
2. 阿呆判断任务类型 → 选择对应 Worker
3. 阿呆读取：
   - `workers/{role}/definition.md`
   - `workers/{role}/log.md`
4. 阿呆组装 Prompt（按标准模板）
5. spawn Worker（run 模式）with 完整 Prompt
6. Worker 执行任务 → 返回结果给阿呆
7. 阿呆更新：
   - `workers/{role}/log.md`（如有完成记录）
   - `MEMORY.md`（汇总工作结果）
8. 阿呆决定下一步（继续 spawn 或自己处理）

### 5.2 Prompt 组装模板

```
你是 {role}-worker（{角色中文名}）。

## 角色定义
{读取 definition.md 的完整内容}

## 工作记忆
{读取 log.md 的核心能力 + 最近3条完成任务记录}

## 当前任务
{具体任务描述}

## 执行要求
1. 先理解需求，再动手
2. 完成过程中如有重要发现或问题，及时汇报
3. 完成后说明：做了什么、结果如何

## 完成后
将执行结果（成功/失败/问题）回复给我。
如需写入工作日志，我会代你更新。
```

## 六、写入规范

### 6.1 Worker 写入规则

| 写入类型 | 是否允许 | 说明 |
|---------|:--------:|------|
| 直接写 log.md | ⚠️ 谨慎 | 仅限追加完成记录，格式必须规范 |
| 读取文件 | ✅ 允许 | 读取 definition.md 和 log.md |
| 创建临时文件 | ✅ 允许 | 测试用，创建后需删除 |
| 删除文件 | ❌ 禁止 | 必须阿呆确认 |
| 修改 definition.md | ❌ 禁止 | 只能阿呆修改 |

### 6.2 阿呆代写规则

以下情况由阿呆统一写入 log.md：
- Worker 完成了重要任务
- Worker 有关键决策或结论
- Worker 提出了重要问题

**阿呆写入格式：**
```
- YYYY-MM-DD HH:MM: {任务描述} | 结果：{成功/失败/问题}
```

## 七、错误处理流程

### 7.1 Worker 执行失败

1. Worker 返回错误结果
2. 阿呆记录问题到 MEMORY.md
3. 阿呆分析失败原因
4. 阿呆决定：重试 / 换 Worker / 自己处理
5. 如重试，更新 log.md 记录失败

### 7.2 Worker 无响应

1. 等待超时（标准 2 分钟）
2. 阿呆记录超时到 MEMORY.md
3. 阿呆重新评估任务
4. 可选择：重新 spawn / 换策略 / 告知用户

---

## 八、阿呆记忆同步规则

每次 Worker 任务完成后，阿呆必须更新：

### 8.1 MEMORY.md（阿呆长期记忆）

在 MEMORY.md 中记录：
- Worker 执行了什么任务
- 结果如何
- 有无遗留问题

### 8.2 workers/{role}/log.md（如需更新）

- 追加完成记录
- 更新核心能力（如有新能力）
- 触发压缩时执行归档

---

## 九、禁止事项

- ❌ Worker 不直接写 log.md（格式可能混乱）
- ❌ Worker 不删除任何文件
- ❌ Worker 不修改 definition.md
- ❌ 阿呆不遗漏更新 MEMORY.md
- ❌ spawn 前阿呆不省略读取 definition.md + log.md

---

## 十、操作授权规则

**AI未经授权不得擅自执行任何操作，包括但不限于：**
- 技能安装/删除
- 仓库文件整理操作
- 敏感Token清除
- SubAgent 重建动作

**所有操作前必须与用户确认并获得明确授权后方可执行。**

---

## 十一、验证清单

每次 spawn Worker 前，阿呆必须确认：

- [ ] 选择了正确的 Worker 角色
- [ ] 读取了 definition.md
- [ ] 读取了 log.md（核心能力 + 最近记录）
- [ ] 按模板组装了完整 Prompt
- [ ] Worker 完成后更新了 MEMORY.md
